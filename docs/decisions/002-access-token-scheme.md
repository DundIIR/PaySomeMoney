# 002. Схема доступа через access_token

**Статус:** принято
**Дата:** 2026-06-11

## Контекст

Статистику должны видеть только оплатившие, и только 24 часа с момента оплаты. Нужен способ связать браузер пользователя с его платежом — без регистрации и логина.

## Решение

При создании платежа `create-payment` генерит уникальный `access_token` (uuid), сохраняет его в записи `payments` и возвращает фронту. Фронт хранит токен в **localStorage** и предъявляет его в `get-stats`. Сервер проверяет: статус `succeeded` и `paid_at` не старше 24 часов.

## Ключевые моменты

### Токен генерится в create-payment, не в вебхуке

Вебхук приходит от ЮKassa сервер-сервер и никак не связан с браузером пользователя — вернуть из него токен юзеру невозможно. А `create-payment` вызывается из браузера, поэтому токен выдаётся сразу в ответе, ещё до оплаты. Право доступа токен получает позже, когда вебхук переведёт платёж в `succeeded`.

### Токен в localStorage, не в URL

URL с токеном попадает в историю браузера, логи nginx, рефереры — это утечка. localStorage этим не страдает. Минус: доступ привязан к конкретному браузеру (другое устройство токен не увидит) — для этого проекта приемлемо.

### RLS без политик вместо Supabase Auth

Полноценная авторизация не нужна — у нас нет пользователей как сущности, есть только платежи. Таблица закрыта RLS целиком, anon не может ничего; вся проверка токена происходит в Edge Functions через service role.

## Последствия

- Очистка localStorage = потеря доступа до конца 24-часового окна. Принимаем.
- Срок доступа проверяется только на сервере (`get-stats`), фронт ничего не решает.
